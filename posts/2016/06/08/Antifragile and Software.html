Antifragile and Software

In reading Antifragile it's seemed that the concept relates to software and programming in many ways, including language design, testing, bug bounties and project management.

First off, I'm only 60 pages in, and the book is over 700 pages, so I'm not working with everything the book has to offer. But let's lead with a vague definition of antifragile:

Antifragility is a property of systems that increase in capability, resilience, or robustness as a result of stressors, shocks, volatility, noise, mistakes, faults, attacks, or failures. It is a concept developed by Professor Nassim Nicholas Taleb in his book, Antifragile.

The first and most obvious software connection is Chaos Monkey, software that will randomly take down one of your servers, raining chaos on your devs and ops. While this is no doubt a traumatic experience for the engineers involved, it's partly responsible for the high quality of service provided by Netflix. But there are other ways the notion of antifragility applies to software.

Static Vs. Dynamic Typing

Python's duck typing is a good example of a dynamically typed programming language, and Haskell's static typing a good example of a statically typed programming language. The Haskell adage, "If it compiles, it likely works", is an indication of the type system's role in correctness, or fragility. Many bugs are attempted to be caught at compile time so they can't influence the program at run time. By contrast Python puts no effort into stopping bugs at compile time.

How does this fit with antifragility? I would say that by the definitions, this puts Python as a fragile language, and Haskell as a robust language. What about antifragile languages? Well if done right, both Python and Haskell can be antifragile with the simple inclusion of some tests.

How so? Well if you encounter a bug you write a test for it. You write code to pass the test, deploy the patch and carry on. Now, with a CI server running before every merge, we've theoretically ruled out the possibility of that bug ever occuring again. The difference is because Python treats program correctness as orthogonal to language design, a Python programmer can't be lazy and rely on the compiler. If they want a program of any meaningful scale they have to add tests. A Haskell programmer has the unfortunate luxury of being able to delay tests, which for an undisciplined programmer can result in buggy programs.

I think the above paragraph works for most pairings of static and dynamic languages. Replace Haskell with Java, and Python with Javascript if you like. The amount of robustness achieved may vary from type system to type system, but the point stands.

Agile Development & Bug Bounties

The other way in which this relates to software development is at the project level. I think it fair to say that a Waterfall design process, with it's more rigid timelines and resource allocations, could be classified as fragile, in contrast to Agile development's more antifragile process. I understand that Agile development varies from organisation to organisation, but in general there is (or should be) an attempt at continual iteration and improvement. The process allows and anticipates flaws being discovered, and uses that as opportunities to improve the product.

Bug bounties are an extreme case of outsourcing the discovery of flaws to the general public. Although they serve a dual purpose of making white hat hacking more favourable than black hat hacking, they could theoretically be used to improve program correctness, not just security (e.g. reward bug reports that get patches merged).

Conclusion

I'm only 60 pages in to Antifragile, but it appears to be a fruitful, if opinionated, read. I look forward to finishing this book.
